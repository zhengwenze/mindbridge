# 变更影响矩阵

矩阵用于路由，不用于强造固定文件名。先识别项目现有的架构、接入、运维、决策和发布文档，再把对应受众更新到位。

## 先删哪些噪音

| 反模式 | 默认处理 |
|---|---|
| “某日 X 上线，详见 Y”塞在主规则顶部 | 现役约束就地融合；纯历史迁 git/changelog/事故文档 |
| 主规则抄完整架构、评分公式或数据流 | 留边界和权威文档指针，详细机制回 docs |
| 多个版本都自称“现役” | 以代码开关、运行态和回退链裁决；历史版显式标回退/退役 |
| 已完成待办仍列开放项 | 核实后删除或改为当前约束，不保留“已完成”流水账 |
| 单次事故长篇常驻 memory | 提炼可复用教训；机制进 docs，过程进 incident/git |
| 多处复制同一规则 | 保留一个权威真身，其他位置放短指针或删除 |
| “保留作历史”但没有真实读者和用途 | 删除；历史需要用途、权威位置和检索入口 |
| Agent 会话残留：一次性计划文档、调试脚本、`xxx_old`/`xxx_backup` 副本 | 有效内容并进正式文档；文件列删除候选，经用户确认后才清 |

## 代码或运行态变化 → 知识面

| 变化 | 要核对的受众面 |
|---|---|
| API / 路由 /协议 | 外部接入合同、架构路由、鉴权、示例、consumer 文档、smoke |
| 环境变量 / provider / model | 示例 env、运行手册、默认值权威、下游配置、成本/回退说明 |
| schema / 存储字段 | 数据模型、迁移与回滚、派生/缓存、读写边界、测试 |
| 用户流程 / 权限 /导航 | README/产品说明、架构权限、运营手册、视觉验收 |
| 评分 /筛选 /聚类 / prompt | 现役算法文档、阈值/开关、回测证据、回退链、展示影响 |
| 服务 /部署 /CDN /边缘 | 运维手册、服务名、marker、cache、回滚、live acceptance |
| 后台任务 /告警 | job 文档、调度、失败收口、告警路由、决策面 |
| 退役 /改名 /下线 | 搜旧 symbol 的非历史引用、consumer、rules、skills 和 memory；保留明确标注的历史证据 |
| 发布流程 /多 Agent 生命周期 | SOP、合并/部署脚本、知识凭证、清场门禁、workspace audit |

## 受众职责

新增或改变一个能力时，至少判断四种职责是否存在；不是要求创建四个固定文件：

1. **怎么用**：README、API/集成指南、示例。
2. **怎么工作**：architecture、schema、状态机、设计边界。
3. **怎么运维**：runbook、环境变量、smoke、回滚、告警。
4. **当前状态/历史**：产品 changelog、release note、incident 或 git；只有项目规则/用户明确要求才写 changelog。

如果同一文档承担多个职责，可以就地更新，不为形式拆文件。

## 记忆路由

| 记忆类型 | 处理 |
|---|---|
| 用户偏好 /反复纠正 | 在平台允许且用户授权时保留短、可执行条目 |
| 系统机制 | 毕业进权威 docs；memory 只留检索指针 |
| 决策事件 | 只保留最终结论和仍有效的为什么；过程归 git/changelog |
| 已完成开放项 | 核实后删，不让它继续冒充计划 |
| 相对时间 | 当前状态改绝对日期；历史叙事中的自然语言不机械替换 |
| 重复或冲突 | 用当前代码/运行态裁决；无法裁决就保留冲突并标 pending |
| 机器生成索引 | 不直接编辑；写规定输入或候选，交给生成流程 |
| 长期未引用 | 结合权威性和复用价值判断；未引用不是单独删除理由 |

## 规则违规 → 动作

| 发现 | 动作 |
|---|---|
| 现场规则要求同源，但软链/导入缺失 | 在授权范围内补齐并验证 realpath/load |
| 两份独立规则内容不同 | 能由现场声明唯一裁决则合并；否则待决 |
| 有可运行代码但缺项目规则 | 默认创建最小规则文件（五要素、精简）；工作空间有模板按模板 |
| ignore 缺安全红线 | 在请求范围和授权内直接补，并验证 tracked secrets |
| 命名违规、根目录乱放 | 只读报告影响；移动/重命名/删除通常待决 |
| 规则引用退役路径/命令 | 当前机器唯一确认且无跨设备疑点时清；否则报告 |
| 同一违规反复出现 | 用 check/hook/test 根治；是否自动实现服从现场授权 |

## 跨项目检查

以下任一为真，就搜索实际 consumer，而不是只改当前项目：

- 共享 API、SDK、域名、认证、schema 或环境变量改变；
- 部署/边缘层改变了下游可见行为；
- 公共 skill、脚本或模板被多个项目引用；
- 当前项目文档明确列出依赖方。

只读发现 consumer 不等于获准编辑它。先说明依赖证据和影响，再按任务与工作空间规则行动。
