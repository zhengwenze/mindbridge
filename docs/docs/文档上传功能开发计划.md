# MindBridge 文档上传功能开发计划

## 一、目标与现状

在管理员端新增 `/admin/docs`“文档管理”页面，实现管理员选择目标知识库后，通过拖拽、文件选择或文件夹选择批量上传文档，并展示逐文件上传与处理进度。

当前项目已有知识库、文档、分片和向量索引模型，以及单文件上传接口，但存在一次性读取文件、缺少 DOCX 解析、无上传进度、无文件夹上传等问题。本次将复用并重构现有链路，不重复建设知识库体系。

本期明确支持：

- `.txt`、`.md`、`.markdown`、`.pdf`、`.docx`
- 单文件最大 50 MB
- 同一知识库、同一相对路径的重复文件返回冲突，不覆盖
- 文件夹上传保留逻辑相对路径
- 搜索、编辑、删除和持久化文档列表不在本期范围

## 二、Git 协作

- 开发前获取最新 `main`，创建并切换到用户指定的 `docs` 分支。
- 远程目前不存在 `docs` 分支，可直接创建。
- 保留当前 `sidebar-placeholder.tsx` 中尚未提交的菜单修改，在 `docs` 分支中整理格式并继续完成。
- 按“后端上传链路”“前端上传页面”“测试与文档”拆分原子提交。
- 验收通过后推送 `origin/docs`，创建以 `main` 为目标分支的 PR，不直接合并主分支。

## 三、实现方案

### 后端上传与解析

- 保留 `POST /api/admin/knowledge-bases/{knowledge_base_id}/documents`，继续使用 `python-multipart` 接收单文件请求。
- multipart 字段包括：
  - `file`：上传文件。
  - `relative_path`：可选逻辑相对路径；旧客户端不传时默认使用文件名。
- 每个文件单独发起请求，便于独立显示进度、失败重试和批量上传部分成功。
- 以 1 MiB 为单位循环读取 `UploadFile` 并写入临时文件，不再调用无参数的 `file.read()`。
- 在读取过程中累计大小，超过 50 MB 立即返回 HTTP 413，并清理临时文件。
- 规范化相对路径，拒绝绝对路径、`..`、空路径和越界字符；物理文件仍使用 UUID 安全命名，相对目录仅作为文档元数据保存。
- 新增 `python-docx`：
  - TXT/Markdown：支持 UTF-8、UTF-8 BOM，无法解码时返回 422。
  - PDF：按页提取文本；加密、损坏或无可提取文本的文件返回 422，不提供 OCR。
  - DOCX：提取正文段落和表格文本，并限制压缩包解压规模，防止异常 DOCX 占用过多资源。
- 使用增量分片器生成带重叠的知识片段，向量生成按批次执行，避免一次性持有全部文件字节或全部 embedding 请求。
- 上传、解析、数据库记录和索引写入采用可补偿流程：任一步失败都回滚新文档、向量和临时文件，使管理员可以直接重试。
- 文档上传失败只影响该文件，不把整个知识库改为错误状态。
- 同一知识库内按规范化相对路径判重；并发上传由数据库唯一约束兜底，返回 HTTP 409。
- 预期错误约定：
  - 409：相对路径重复。
  - 413：超过 50 MB。
  - 415：格式不支持。
  - 422：文件损坏、编码错误或没有可索引文本。
  - 503/500：索引依赖不可用或内部处理失败。

### 数据库与接口类型

- 为 `knowledge_documents` 增加非空 `relative_path` 字段，已有数据回填为原 `file_name`。
- 将唯一约束从 `(knowledge_base_id, file_name)` 调整为 `(knowledge_base_id, relative_path)`。
- `file_name` 保存 basename，`relative_path` 保存最长 512 字符的逻辑路径。
- 扩展知识片段的 `source` 长度并使用相对路径作为来源，确保检索结果能够定位文件夹中的文档。
- 上传响应保持现有字段兼容，并增加：
  - `knowledgeBaseId`
  - `relativePath`
  - `fileSize`
- 配置中增加上传大小、读取块大小和 DOCX 解压上限，默认分别为 50 MB、1 MiB 和 200 MB。

### 管理员前端

- 新增 `/admin/docs` 页面，并在管理员侧边栏加入“文档管理”菜单，使用独立文档/文件夹图标。
- 页面提供可搜索的目标知识库选择器，仅允许选择状态为 `active` 的知识库；一次上传队列只对应一个知识库。
- 使用 Ant Design 上传组件实现：
  - 拖拽多个文件。
  - 点击选择多个文件。
  - 独立的“选择文件夹”入口。
  - 读取浏览器 `webkitRelativePath` 并提交给后端。
- 客户端预检文件扩展名、大小和队列内重复相对路径；服务端继续执行最终校验。
- 上传队列默认最多并发 2 个文件，上传期间锁定目标知识库。
- 每个文件显示“等待、上传中、解析入库中、成功、失败”状态：
  - Axios `onUploadProgress` 显示真实字节上传百分比。
  - 字节达到 100% 后切换为“解析入库中”，直到接口返回。
  - 失败项显示后端原因并支持单文件重试；等待项可以移除。
- 批次结束显示成功、失败数量汇总；某个文件失败不影响其他文件。
- 使用上传专用长超时配置，避免全局 20 秒接口超时中断大文件处理。
- 本期成功记录只保留在当前上传队列中，刷新页面后不展示文档历史；后续搜索、编辑、删除功能再引入文档列表接口。

## 四、测试与验收

### 后端测试

- 五种受支持扩展均可解析、分片并写入指定知识库。
- 未登录和非管理员请求分别返回 401/403。
- 验证服务端始终以不超过 1 MiB 的块读取上传内容。
- 50 MB 文件允许上传，超过 1 字节返回 413且无临时文件和数据库残留。
- 不支持格式、损坏 PDF/DOCX、空文档、非 UTF-8 文本返回明确错误。
- `../`、绝对路径和非法相对路径被拒绝。
- 同一路径重复上传返回 409；`A/说明.txt` 与 `B/说明.txt` 可以共存。
- 解析或向量索引失败后完整回滚，并可再次上传。
- 数据库迁移正确回填历史文档，升级和回滚均可执行。
- 上传完成后，通过知识库检索只能从目标知识库召回新文档内容。

### 前端与集成验证

- `/admin/docs` 仅管理员可访问，菜单选中状态正确。
- 文件拖拽、多文件选择和文件夹选择均能建立正确队列。
- 相对路径、客户端格式限制、50 MB 限制、并发数和失败重试行为正确。
- 上传阶段显示百分比，上传完成后显示解析状态，最终显示入库结果。
- 批量上传部分失败时，其余文件仍可成功。
- 执行 Python 测试、数据库集成测试、前端类型检查、Lint 和生产构建。
- 使用 Docker 环境进行一次完整验收：创建或选择知识库，上传五种格式及嵌套文件夹，确认数据库记录、文件存储、分片数量和 Chroma 检索结果一致。

## 五、默认约定

- “重复文件”以知识库 ID 加规范化相对路径判定，大小写遵循当前 MySQL 排序规则。
- 文件夹层级保存在数据库和检索来源中，不在磁盘上创建用户提供的目录结构。
- PDF 首版只支持可提取文本的数字 PDF，不支持扫描件 OCR。
- 一个批次只能上传到一个知识库；需要上传到多个知识库时分别建立批次。
- 本 PR 同步更新知识库开发文档、上传接口说明和相关配置说明。
