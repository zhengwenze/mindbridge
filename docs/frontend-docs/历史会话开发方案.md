# MindBridge 学生端历史会话开发方案

> 状态（2026-07-21）：已实施，保留为历史方案。现役实现位于 `app/api/routes.py`、`frontend/src/features/chat/` 和 `frontend/src/app/student/history/`。

## Summary

在现有学生端增加“历史会话”菜单和历史会话工作区。学生可以查看自己的历史会话，点击某条会话后加载完整聊天内容，并继续在原会话中发送消息。

现有数据结构已经满足需求：

- MySQL `chat_sessions` 保存会话标题、所属学生和更新时间。
- MySQL `chat_messages` 保存完整聊天内容。
- Redis 保存短期对话记忆，仅用于提升智能体上下文读取速度。
- 继续聊天时，后端已经通过 `sessionId` 校验当前学生身份，并复用原会话记忆。

因此不需要新增数据库表或修改现有表结构。

## Backend changes

### 1. 增加学生会话列表接口

新增：

```http
GET /api/student/sessions
```

仅允许已登录学生访问，返回当前用户自己的会话列表，按 `updated_at` 倒序排列。

建议响应：

```json
[
  {
    "sessionId": "abc123",
    "title": "我最近压力很大，晚上总是睡不着。",
    "lastMessage": "可以先从今晚的睡前安排开始调整……",
    "messageCount": 4,
    "createdAt": "2026-07-10T10:00:00",
    "updatedAt": "2026-07-10T10:15:00"
  }
]
```

实现要求：

- 必须按 `ChatSession.user_id == current_user.id` 过滤。
- 不允许学生通过修改 `sessionId` 读取其他学生的会话。
- 排序使用 `ChatSession.updated_at.desc()`。
- `lastMessage` 使用该会话最后一条消息。
- 当前会话创建逻辑保持不变，首次发送消息时自动创建会话。
- 第一版展示该学生全部历史会话，暂不加入分页和搜索。

### 2. 增加学生会话详情接口

新增：

```http
GET /api/student/sessions/{session_id}
```

返回完整会话内容：

```json
{
  "sessionId": "abc123",
  "title": "我最近压力很大，晚上总是睡不着。",
  "messages": [
    {
      "id": 1,
      "role": "user",
      "content": "我最近压力很大",
      "createdAt": "2026-07-10T10:00:00"
    },
    {
      "id": 2,
      "role": "assistant",
      "content": "听起来你最近承受了不少压力……",
      "createdAt": "2026-07-10T10:00:05"
    }
  ]
}
```

实现要求：

- 查询会话时同时校验 `public_id` 和当前用户 ID。
- 会话不存在或不属于当前用户时统一返回 404，避免泄露其他用户会话是否存在。
- 消息按 `created_at.asc()` 返回。
- 复用现有会话详情的数据结构，但不要直接复用管理员接口的权限逻辑。
- 现有 `/api/admin/conversations/{session_id}` 保持不变。

### 3. 继续聊天逻辑保持复用

前端加载历史消息后，发送新消息时继续调用：

```http
POST /api/chat/stream
```

请求中的 `sessionId` 使用已选中的历史会话 ID：

```json
{
  "sessionId": "abc123",
  "message": "那我今晚应该怎么做？"
}
```

后端现有逻辑已经支持：

- 根据 `sessionId` 找回会话。
- 校验会话属于当前用户。
- 优先从 Redis 读取短期记忆。
- Redis 不可用时从 MySQL `chat_messages` 回退读取。
- 新消息继续写入 MySQL 和 Redis。
- 更新会话 `updated_at`。
- 保持原有风险评估、知识检索、多智能体和后台报告流程。

不修改 `RedisShortTermMemoryStore` 的数据结构，也不把 Redis 当作历史会话列表的数据源。

## Frontend changes

### 1. 增加学生端菜单项

在学生侧边栏增加：

```text
心理咨询
历史会话
```

建议使用：

- 心理咨询：`/student`
- 历史会话：`/student/history`

“心理咨询”用于开始新会话；“历史会话”用于查看和恢复旧会话。

### 2. 新增历史会话 API 和类型

新增学生端 API 方法：

```ts
fetchStudentSessions(): Promise<StudentSessionSummary[]>
fetchStudentSession(sessionId: string): Promise<StudentSessionDetail>
```

新增类型：

```ts
interface StudentSessionSummary {
  sessionId: string;
  title: string;
  lastMessage: string;
  messageCount: number;
  createdAt: string;
  updatedAt: string;
}

interface StudentSessionDetail {
  sessionId: string;
  title: string;
  messages: ConversationMessage[];
}
```

统一使用现有 `apiClient`，自动复用 Basic Auth 和 401/403 处理。

### 3. 抽取可复用的聊天工作区

当前 `StudentChat` 同时负责：

- 消息展示
- 输入
- 流式响应
- 新会话
- sessionId 管理

建议将聊天能力抽取为可复用组件或 Hook，使“新会话”和“历史会话”共用同一套续聊逻辑。

需要支持以下状态：

- `messages`
- `sessionId`
- `sessionTitle`
- `streaming`
- `sessionStatus`
- 加载历史会话状态
- 历史会话加载失败状态

加载历史会话后：

1. 根据接口消息生成前端 `Message[]`。
2. 保留接口返回的 `sessionId`。
3. 后续发送消息时将该 ID 传给 `/api/chat/stream`。
4. 流式回复完成后更新当前聊天内容。
5. 返回历史列表时刷新对应会话的 `updatedAt`、摘要和消息数。

### 4. 新增历史会话工作区

建议页面结构：

```text
┌──────────────┬────────────────────────────┐
│ 历史会话列表  │ 当前会话聊天窗口             │
│              │                            │
│ 今天          │ 会话标题                   │
│  期末压力     │ 历史消息                   │
│ 睡眠问题      │                            │
│              │ 输入框 + 发送               │
└──────────────┴────────────────────────────┘
```

交互规则：

- 进入 `/student/history` 后加载会话列表。
- 默认自动选中最近更新的一条会话。
- 点击列表项后加载该会话完整消息。
- 加载期间显示 Skeleton 或 Loading。
- 没有历史会话时显示空状态，并提供“开始新的心理咨询”按钮。
- 当前会话发送中时，禁止切换其他会话。
- 点击“新会话”后清空当前消息和 `sessionId`，但仍停留在聊天工作区。
- 会话列表使用标题、最近更新时间、最后一条消息摘要。
- 移动端改为上下布局或抽屉式会话列表，避免挤压聊天输入区。

### 5. 路由与刷新行为

推荐使用：

```text
/student/history?sessionId=<session_id>
```

行为：

- 不带 `sessionId`：加载列表并默认选中最近会话。
- 带 `sessionId`：加载指定会话。
- 浏览器刷新后仍能恢复当前会话。
- 非法或无权限的 `sessionId`：显示“会话不存在或无法访问”，不暴露详细权限信息。
- 从“心理咨询”进入时默认创建空白新会话，不自动创建数据库记录，直到用户发送第一条消息。

## Testing plan

### Backend tests

新增接口测试覆盖：

- 学生可以获取自己的历史会话。
- 会话按照最近更新时间倒序排列。
- 会话摘要和消息数量正确。
- 学生无法读取其他学生的会话。
- 不存在的会话返回 404。
- 会话详情消息按时间正序返回。
- 使用历史 `sessionId` 继续聊天后，新消息仍写入原会话。
- Redis 可用时使用 Redis 上下文。
- Redis 不可用时从 MySQL 消息回退读取。
- 管理员接口权限不受影响。
- 管理员不能调用学生聊天接口的既有规则保持不变。

### Frontend tests

验证：

- 学生侧边栏显示“心理咨询”和“历史会话”。
- 点击“历史会话”后进入正确页面。
- 历史列表能够正确加载。
- 点击会话后聊天消息正确展示。
- 历史会话发送新消息时携带正确 `sessionId`。
- 新消息流式展示在原会话末尾。
- 发送期间不能切换会话或重复发送。
- 空列表、加载失败、详情加载失败状态正常。
- 刷新带 `sessionId` 的页面后仍能恢复会话。
- 移动端列表和输入区不发生遮挡。

### 验收标准

- 学生只能看到自己的会话。
- 点击历史会话后能看到完整历史聊天内容。
- 在历史会话中发送消息后，智能体能够理解原有上下文。
- 新消息不会创建错误的新会话。
- 刷新页面不会丢失当前选中的会话。
- Redis 清空或不可用时，历史消息和续聊能力仍可正常工作。
- 学生端不展示风险等级、情绪评分、后台报告或 Agent Trace。

## Assumptions

- 第一版展示当前学生全部历史会话，不做分页、搜索、删除和重命名。
- 历史列表默认按最近更新时间倒序。
- 默认自动打开最近更新的会话。
- 会话标题继续使用现有逻辑，即首次用户输入前 36 个字符。
- 继续聊天沿用现有 `/api/chat/stream`，不另建聊天接口。
- 不修改数据库表结构和 Redis 存储格式。
