# 多知识库管理重构方案

> 状态（2026-07-21）：已实施，保留为历史方案。现役运行与接口说明以 [`knowledge-base.md`](knowledge-base.md) 为准，数据迁移以 `migrations/versions/20260713_0002_multi_knowledge_bases.py` 及后续迁移为准。

## 摘要

当前仓库没有 Alembic；应用启动时以 `Base.metadata.create_all()` 建表。现有 RAG 使用单一 `knowledge_chunks` 表和 `mindbridge_knowledge` collection，前端只支持全局上传、重建、备份。

本次将建立“知识库 → 文档 → 片段”的 MySQL 数据模型；每个知识库使用不可变的 `mindbridge_kb_<id>` Chroma collection；统一采用 `settings.embedding_model` / `EMBEDDING_MODEL=qwen3-embedding:0.6b`，不提供知识库级模型配置。

已确认策略：

- 旧数据统一迁入“心理健康基础知识库”；先备份原记录和 Chroma 持久化目录，再重建新 collection，旧全局 collection 暂不删除。
- 聊天采用轻量规则路由：咨询默认基础库；风险检测与人工升级链路保持不变，并辅助危机库；命中明确政策关键词时辅助政策库。
- 物理删除不支持 `force`；存在外部引用一律返回 409。

## 数据库、迁移与配置

- 引入 Alembic，新增基线迁移覆盖现有 SQLAlchemy 表，再新增多知识库迁移；Docker entrypoint 在 MySQL 就绪后执行 `alembic upgrade head`，应用启动不再调用 `create_all()`。
- 增加 `knowledge_bases`：
  - `id`、`name`、`description`、不可变且唯一的 `collection_name`、`status`、`created_by`、时间字段、`deleted_at`。
  - 使用 MySQL 生成列/唯一索引保证“未删除名称唯一”；逻辑删除后允许同名新建，并为新库生成新的 collection。
  - `created_by` 外键关联 `user_accounts.id`；迁移期间无法关联管理员的历史种子记录允许暂时为空，启动 seed 在管理员存在后补齐。
- 增加 `knowledge_documents`：
  - 归属知识库、文件名、文件类型、文件大小、安全的本地存储路径、索引状态、错误信息、创建/删除时间。
- 改造 `knowledge_chunks`：
  - 增加 `knowledge_base_id` 与 `document_id` 外键，保留现有 `source` 以兼容历史数据和来源展示。
  - 新增 `knowledge_base_id`、`document_id`、复合 `(knowledge_base_id, document_id)` 索引。
- 增加轻量 `knowledge_base_operation_logs`，持久记录创建、编辑、逻辑删除、恢复、物理删除和重建的管理员、目标库、结果与脱敏错误摘要；不复用仅面向工具作业的 `tool_audit_records`。
- 配置收敛为全局 `embedding_model`，默认和 Docker/.env.example 均为 `EMBEDDING_MODEL=qwen3-embedding:0.6b`；EmbeddingService 仅通过 Ollama `/api/embed` 调用该全局模型。移除新链路对 `OPENAI_EMBEDDING_MODEL` 和 `CHROMA_COLLECTION_NAME` 的依赖。
- 默认 seed 幂等创建三个知识库；内置 `app/knowledge/*.md` 统一同步到“心理健康基础知识库”。

## 旧数据迁移与后端实现

- 提供一次性迁移命令：
  1. 导出旧 `knowledge_chunks` 记录为 JSON，并快照 `data/chroma`。
  2. 对已存在、未受 Alembic 管理的数据库执行明确的 baseline adoption，再升级至最新迁移。
  3. 创建默认三库；为每个旧 `source` 建立基础库文档，将全部旧片段回填到对应 `document_id` 与基础库 ID。
  4. 只为基础库重新索引；校验 MySQL 片段数与新 collection 向量数一致。
  5. 记录旧 collection 名称与保留期限，不查询、不删除旧 collection。
- 拆分现有 `KnowledgeService`：
  - KnowledgeBaseService：创建、列表、详情、编辑、状态变迁、逻辑删除、恢复、purge、引用检查与操作日志。
  - KnowledgeDocumentService：安全保存文件、解析 PDF/Markdown/TXT、同库同名文件替换、文档/片段写入及索引状态更新。
  - VectorStoreService：集中生成 collection 名、创建/存在检查/计数/删除 collection、指定可信 collection 的向量写入和查询。
  - EmbeddingService：统一生成 qwen3 向量，禁止客户端传入模型或 collection 名。
- 创建知识库先持久化为 `indexing`，使用 ID 生成 collection 并创建；成功才转为 `active`，失败持久化为 `error`，不报告为正常。
- 重建使用“先 upsert、验证、后删除过期向量”的同步流程，绝不先清空旧 collection；失败保留现有向量并将知识库标为 `error`。
- 禁用、删除、索引中或错误状态的知识库禁止上传和普通检索。恢复逻辑删除库时，若 collection 缺失则转为 `error`，要求管理员重建后再启用。
- purge 仅允许已逻辑删除的库：先检查 `AgentRunTrace.retrieved_knowledge_json` 中的知识库引用；存在引用返回 409。无引用时删除指定 collection、文档文件、片段、文档和知识库记录；任何 Chroma/数据库失败保留失败状态和操作日志，允许管理员重试，不伪装成跨系统事务成功。
- RAG 检索改为显式接收经服务端校验的知识库 ID 列表，只访问其数据库片段和 collection；检索结果增加知识库 ID、知识库名称、文档名。所有现有 runtime、event-driven runtime、autonomous runtime、LangGraph 路径和 RAG 评测均改用该接口。

## API 与前端

- 新增管理员接口：
  - `POST /api/admin/knowledge-bases`
  - `GET /api/admin/knowledge-bases?name&status&created_from&created_to&page&page_size&include_deleted`
  - `GET /api/admin/knowledge-bases/{id}`
  - `PATCH /api/admin/knowledge-bases/{id}`
  - `DELETE /api/admin/knowledge-bases/{id}`（逻辑删除）
  - `POST /api/admin/knowledge-bases/{id}/restore`
  - `DELETE /api/admin/knowledge-bases/{id}/purge`
  - `POST /api/admin/knowledge-bases/{id}/documents`
  - `GET /api/admin/knowledge-bases/{id}/status`
  - `POST /api/admin/knowledge-bases/{id}/rebuild`
- 列表统一返回分页数据、文档数、片段数、向量数/collection 状态，采用聚合查询，避免 N+1。
- `PATCH` 仅接受名称、描述以及管理员可操作的 `active`/`disabled` 状态；`indexing`、`error`、`deleted` 为系统状态，不能由普通编辑伪造。
- 重写 `/admin/knowledge` 的 API、类型、React Query hooks 和 `KnowledgeBasePanel`：
  - 顶部提供新建、名称搜索、状态筛选、日期范围、重置与刷新。
  - Ant Design Table 展示真实服务端分页、描述、文档数、片段/向量数、状态、创建/更新时间和操作。
  - 创建/编辑 Drawer 或 Modal 只显示全局 Embedding 模型，不提供模型下拉框；collection 名和时间只读。
  - 点击“管理文档”进入当前库的 Drawer：显示知识库状态，上传明确绑定当前 ID，重建仅作用于当前库。
  - 逻辑删除使用二次确认；已删除项才显示 purge，要求输入知识库名称确认，并展示后端 409 引用阻塞信息。
  - 移除无知识库归属的旧上传、全局重建及旧全局 collection 状态接口调用；保留 Chroma 存储快照时明确标注“全局存储快照，不是单库导出”。
