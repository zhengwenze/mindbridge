# MindBridge 前端设计系统

> 本文件是 MindBridge 前端视觉与交互设计的唯一事实来源。新增页面、修改布局或评审 UI 前必须先阅读本文件；未经明确确认，不要创建与这里冲突的第二套风格。

## 1. 产品与设计定位

- **产品**：面向高校学生与心理支持管理人员的心理健康智能体 Web 应用。
- **学生端目标**：让学生愿意开口、安心表达，并持续获得清晰、低压力的支持。
- **管理端目标**：帮助管理员快速扫描风险、追踪个案、回溯报告并完成处置闭环。
- **项目形态**：学生陪伴工作台 + 数据密集型管理后台。
- **技术基线**：Next.js、React、TypeScript、Ant Design、Tailwind CSS。

MindBridge 不是营销网站，也不是普通 CRUD 模板。页面第一屏应直接进入工作，不设置 Hero、大段欢迎语或重复页面介绍。

## 2. 核心理念

### 2.1 五个关键词

1. **可信**：结构稳定、状态明确、反馈及时，不用视觉噱头掩盖信息。
2. **温和**：学生端低刺激、非医疗化、不制造紧张感。
3. **高效**：管理端信息密度高，关键状态可快速扫描和操作。
4. **克制**：少装饰、少层级、少重复，组件必须为任务服务。
5. **一致**：相同信息使用相同组件、颜色、间距和交互方式。

### 2.2 决策优先级

视觉决策冲突时，按以下顺序判断：

```text
心理安全与隐私
> 信息清晰与任务效率
> 跨页面一致性
> 响应式与可访问性
> 视觉装饰
```

### 2.3 减法优先

- 全局 Header 或 Sidebar 已表达页面位置时，不再重复展示面包屑、大标题和说明段。
- 卡片标题只描述卡片任务，不复述页面名称。
- 用户无需据此行动的解释文字应删除，或移动到 Tooltip、帮助提示、空状态中。
- 同一视区最多一个主操作。
- 不使用卡片套卡片制造层级；优先用间距、分割线和标题组织内容。

## 3. 双工作区性格

### 3.1 学生端：安静的陪伴工作台

关键词：温和、安静、陪伴、安全、低干扰。

- 视觉焦点是对话内容和输入框，不是模型、风险或运营指标。
- 使用浅灰白背景、白色内容面和少量蓝绿色强调。
- 快捷表达帮助学生开口，文案应像邀请，不像诊断问卷。
- 学生消息靠右并使用浅品牌色；MindBridge 消息靠左并使用白色。
- 保持舒适行高，长回复可读，引用和代码不破坏聊天宽度。
- 页面四周使用统一 16px 间距，聊天卡片内部高频区域使用 16px 内边距。
- 禁止展示风险等级、情绪分数、置信度、后台报告、Agent trace 和工具审计。
- 禁止使用恐吓、诊断、过度承诺或替代专业咨询的表达。

### 3.2 管理端：紧凑的处置工作台

关键词：高密度、可扫描、可追踪、可审计、可闭环。

- 页面直接进入指标、筛选、列表或业务卡片，不显示重复页头。
- 优先突出风险等级、状态、负责人、更新时间、失败原因和关联链路。
- 表格、筛选栏和卡片保持紧凑；装饰不能挤占业务信息。
- 列表详情优先用 Drawer，避免离开当前上下文。
- 删除、批量删除、重建索引等高风险操作必须确认。
- 红色只用于真实高风险、错误和危险操作，不能作为普通强调色。

## 4. 视觉语言

### 4.1 美学方向

- **方向**：克制的专业工作台。
- **装饰等级**：最小到适度；允许轻量状态色和少量指标卡背景形状。
- **布局方式**：严格网格、左对齐、清晰边界、稳定尺寸。
- **色彩方式**：一套品牌色 + 冷灰中性色 + 语义色。
- **动效方式**：仅保留帮助理解状态变化的功能性动效。

避免典型模板感：紫色渐变、彩色发光、统一大圆角、图标圆球矩阵、装饰性浮动元素、营销式三列功能卡。

### 4.2 字体

使用系统字体栈，优先保证中文渲染稳定和加载速度：

```css
-apple-system,
BlinkMacSystemFont,
"Segoe UI",
"PingFang SC",
"Hiragino Sans GB",
"Microsoft YaHei",
sans-serif
```

代码和结构化文本使用：

```css
ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, monospace
```

### 4.3 字号与层级

| 角色       | 建议字号 | 使用规则                            |
| ---------- | -------: | ----------------------------------- |
| 工作台标题 |     16px | 只用于全局 Header，不在管理页面重复 |
| 卡片主标题 |  16–20px | 默认 16–18px；强业务区可用 20px     |
| 正文与表单 |  14–16px | 中文正文优先保证可读性              |
| 表格正文   |  13–14px | 信息密集但不可小于 13px             |
| 辅助信息   |  12–13px | 时间、提示、次级元数据              |
| 标签       |     12px | 状态标签必须同时包含文字            |
| 指标数字   |     30px | 仅用于管理概览统计卡                |

规则：

- 不使用超大页面标题。
- 同一层级字号和字重必须一致。
- 正文行高约 1.5；长回复使用更舒适的 1.7–1.8。
- 不使用负字距或随视口连续缩放的字体。
- 数据列和统计数字应使用等宽数字特性或稳定列宽。

## 5. 色彩 Token

所有控件颜色优先来自 `frontend/src/lib/theme/antd-theme.ts`，全局表面色来自 `frontend/src/app/globals.css`。

### 5.1 品牌与中性色

| Token             | 值        | 用途                             |
| ----------------- | --------- | -------------------------------- |
| `brand` / Primary | `#12847f` | 主按钮、链接、选中状态、品牌强调 |
| `brand-dark`      | `#0f766e` | Logo、深色品牌强调、图标         |
| `page-bg`         | `#f8fafc` | 页面和工作区背景                 |
| `surface`         | `#ffffff` | 卡片、Header、Sidebar、输入面    |
| `text-primary`    | `#0f172a` | 标题、正文、关键数据             |
| `text-secondary`  | `#64748b` | 辅助信息、时间和说明             |
| `border-subtle`   | `#d9e2e7` | 卡片、控件和内容边界             |
| `surface-muted`   | `#f1f5f9` | 代码、图标底、次级内容面         |

### 5.2 语义色

| 语义    | 值        | 用途                       |
| ------- | --------- | -------------------------- |
| Info    | `#2563eb` | 系统信息、处理中提示       |
| Success | `#16a34a` | 正常、成功、LOW            |
| Warning | `#f59e0b` | 等待、处理中、MEDIUM       |
| Error   | `#dc2626` | 错误、失败、HIGH、危险操作 |

语义色必须和文字、图标或状态名称一起出现，不允许只靠颜色传递含义。学生端不展示后台风险色。

## 6. 间距与尺寸

### 6.1 4px 基础尺度

| Token |   值 | 典型用途                         |
| ----- | ---: | -------------------------------- |
| `2xs` |  4px | 图标与文字、极小修正             |
| `xs`  |  8px | 标签间距、紧凑操作组             |
| `sm`  | 12px | 表单项、筛选控件、聊天区块间距   |
| `md`  | 16px | 页面边距、卡片间距、聊天卡内边距 |
| `lg`  | 20px | 管理端主要区块间距               |
| `xl`  | 24px | 默认管理卡片内边距、较大分区     |
| `2xl` | 32px | 仅用于明确的大区块，不作默认值   |

使用原则：

- 相邻同级卡片默认 `16px`，复杂管理区块可用 `20px`。
- 同一卡片内的相关控件优先 `8px` 或 `12px`。
- 页面外边距和卡片内边距必须成体系，不能同时出现多套无意图的数值。
- 学生聊天页统一 `16px` 页面边距；不要再使用宽屏限宽造成巨大空白。
- 管理卡片可保留 Ant Design 默认 `24px`，高频操作卡可明确覆盖为 `16px`。

### 6.2 控件尺寸

- 默认控件高度：`36px`。
- 菜单项高度：`40px`。
- 全局 Header 高度：`64px`。
- Sidebar：展开 `180px`，折叠 `80px`。
- Button 圆角：`6px`。
- Card 圆角：`8px`。
- 消息气泡与内容块圆角：`6px`。
- 不要把所有组件改成大圆角或胶囊形。

## 7. 页面布局

### 7.1 应用骨架

```text
AppShell
├── Sidebar：主导航与工作区身份
└── Main
    ├── GlobalHeader：折叠、工作台名称、系统状态、用户、退出
    └── Content：页面业务内容
```

- Header 和 Sidebar 是稳定导航层，不随业务页面重复。
- 主内容区必须 `min-width: 0`，避免表格或长文本撑破布局。
- 桌面端 Sidebar 固定；窄屏必须提供明确的导航替代方案，不可只隐藏后结束。

### 7.2 管理页面

管理员页面默认结构：

```text
PageContainer(hideHeader)
├── Toolbar / Filter（需要时）
├── Main Cards / Table
└── Drawer / Modal
```

- 不显示 `MindBridge / 页面名` 面包屑、大页面标题或说明段。
- `PageContainer` 提供 `max-w-7xl` 与响应式水平边距。
- 页面内容第一层直接是业务组件，卡片标题承担局部信息层级。
- 列表页优先：筛选栏 → 错误反馈 → 表格 → 分页。

### 7.3 学生页面

学生聊天页默认结构：

```text
main(w-full, p-4)
└── Chat Card(p-4)
    ├── Quick Prompts + Session Status
    ├── Message Area
    └── Textarea + Actions
```

- 页面和主卡四周均采用明确、均衡的 16px 边距。
- 消息区吸收剩余高度，输入区始终清晰可见。
- 历史会话页在小屏纵向排列，在 `lg` 以上使用列表 + 对话双栏。

## 8. 高频组件模式

### 8.1 筛选工具栏

桌面管理端优先单行 Grid，不使用会自动换行的 `flex-wrap`：

```tsx
<div
  className="grid w-full min-w-0 items-end gap-2"
  style={{
    gridTemplateColumns: "minmax(0, 1fr) 128px 128px 112px auto auto auto",
  }}
>
  {/* 搜索、开始日期、结束日期、状态、重置、刷新、主操作 */}
</div>
```

- 搜索框使用 `minmax(0, 1fr)` 吸收剩余宽度，并设置 `w-full min-w-0`。
- 日期和下拉框使用稳定固定列宽。
- 按钮使用 `auto`，必要时 `shrink-0 px-3`。
- 所有控件必须保持同一水平线且不超过卡片或页面宽度。
- 窄屏确实无法容纳时，必须定义明确断点策略，不能让布局随机折行。

### 8.2 Card

- Card 用于有明确边界的任务、列表、指标或设置组。
- 卡片标题简短直接；说明只在帮助理解或操作时保留。
- 卡片内不再嵌套同等级 Card。
- 指标卡允许轻量、低透明度状态色装饰，但信息仍是第一视觉层。
- 加载、错误和空状态应在相同卡片边界内显示，避免页面跳动。

### 8.3 Button

- 主操作：`type="primary"`，同一操作区最多一个。
- 次操作：默认 Button。
- 危险操作：`danger`，并提供 Popconfirm 或 Modal。
- 图标按钮必须有可访问名称和 Tooltip。
- 禁用态必须说明原因或由上下文清晰表达。

### 8.4 Table 与列表

- 必须处理 loading、empty、error 和 pagination。
- 长文本使用省略、换行或 Drawer，不允许撑破列宽。
- 时间格式和状态标签保持统一。
- 固定操作列时设置稳定宽度；表格可在管理端横向滚动。
- 空状态要说明当前为何为空，适合时给出下一步操作。

### 8.5 Tag 与状态

- 风险：LOW=`success`、MEDIUM=`warning`、HIGH=`error`。
- 任务：处理中=`processing`、成功=`success`、失败=`error`。
- 未知状态使用 `default`，不可假装成功。
- 标签文本优先使用用户理解的中文；协议字段可在技术详情中保留英文。

### 8.6 Drawer 与 Modal

- Drawer：长详情、会话档案、拆分预览、Agent trace，保留列表上下文。
- Modal：删除确认、重建确认、短表单和关键选择。
- 复杂详情不要塞进小 Modal；短确认不要打开完整页面。

### 8.7 Form、Upload 与反馈

- Ant Design Form 负责校验和提交状态。
- 提交时禁用重复操作，失败信息必须具体，成功后给出即时反馈。
- 上传展示队列、进度/状态、成功结果、失败原因和重试入口。
- 重置筛选必须同时清理请求状态和受控输入显示值。

## 9. Ant Design 与 Tailwind 分工

### Ant Design 负责

- Button、Input、Select、Form、Table、Card、Tag、Drawer、Modal、Alert、Empty。
- 主题色、控件状态、交互反馈和可访问性基础。

### Tailwind 负责

- Flex/Grid 布局、间距、宽高、对齐、响应式和少量表面辅助样式。

### 边界规则

- 优先调整 `antdTheme` token，不逐页覆盖主色、圆角和控件高度。
- 只在业务布局确有需要时覆盖 Card body padding 等局部尺寸。
- 不深度重写 AntD Table、Modal 或 Form 的内部 DOM。
- 不引入第二套组件库解决 AntD 已覆盖的问题。

## 10. 状态、内容与心理安全

### 10.1 所有数据组件必须覆盖

| 状态    | 用户应看到什么                           |
| ------- | ---------------------------------------- |
| Loading | 与最终结构相近的 Skeleton 或明确加载文案 |
| Empty   | 原因、当前状态，适合时提供下一步操作     |
| Error   | 具体失败内容与重试入口                   |
| Success | 结果确认；无需长期占据页面               |
| Partial | 已成功部分、失败部分以及可恢复方式       |

### 10.2 文案规则

- 中文优先，短句、主动语态、具体动词。
- 按钮写实际动作，如“重新索引”“批量删除”，不写含糊的“确定”“继续”。
- 说明文字必须帮助用户判断或行动，否则删除。
- 加载文案使用中文省略号“…”；不要堆叠技术术语。
- 学生端保持支持性语言，不做诊断、不贴标签、不暴露后台判断。

## 11. 响应式与可访问性

### 11.1 断点策略

- `sm`（640px）：基础表单和双列卡片开始展开。
- `md`（768px）：桌面 Sidebar 出现。
- `lg`（1024px）：学生历史页切换双栏，系统状态信息可展示。
- `xl`（1280px）：管理概览多列指标和复杂双栏布局展开。

管理端桌面优先；学生端必须保证移动端可完成核心聊天流程。

### 11.2 可访问性底线

- 主要交互支持键盘操作和可见焦点。
- 图标按钮必须有 `aria-label` 与 Tooltip。
- 风险与状态不能只靠颜色。
- 交互目标移动端建议至少 44×44px；桌面紧凑控件不得牺牲可辨识性。
- 文本与背景满足 WCAG AA：正文至少 4.5:1，大文本至少 3:1。
- 长名称、路径和错误信息必须支持截断、换行或完整 Tooltip。
- Drawer/Modal 打开后焦点正确进入，关闭后回到触发元素。

## 12. 动效

- **原则**：minimal-functional，只为解释状态变化。
- 微交互：`100–150ms`；普通过渡：`150–250ms`。
- 进入用 ease-out，退出用 ease-in，位置变化用 ease-in-out。
- 允许：Sidebar 宽度、Drawer、加载、消息流式输出、按钮反馈。
- 禁止：大面积入场、漂浮装饰、无意义循环动画、`transition: all`。
- 必须尊重 `prefers-reduced-motion`。

## 13. 快速开发模板

### 13.1 新管理页面

```tsx
export default function AdminFeaturePage() {
  return (
    <PageContainer title="功能名称" hideHeader>
      <div className="grid gap-4">
        <FeatureToolbar />
        <FeatureContent />
      </div>
    </PageContainer>
  );
}
```

### 13.2 新学生工作区

```tsx
export default function StudentFeaturePage() {
  return (
    <main className="flex min-h-[calc(100vh-64px)] w-full p-4">
      <FeaturePanel />
    </main>
  );
}
```

### 13.3 提交前检查

1. 页面是否直接进入任务，而不是先读一段介绍？
2. 学生端是否隐藏了后台评估和风险信息？
3. 颜色、间距、圆角是否使用现有 token？
4. 筛选栏是否对齐、同高、受页面宽度约束？
5. 是否覆盖 loading、empty、error、success？
6. 危险操作是否确认，按钮是否防重复提交？
7. 长文本、窄屏、零数据和接口失败是否可用？
8. 是否通过测试、TypeScript、ESLint 和生产构建？

## 14. 禁止模式

- 重复页面面包屑、大标题和说明文案。
- 宽屏限宽导致工作区两侧出现无意义巨幅空白。
- `flex-wrap` 让桌面筛选栏随机折行。
- 不同页面随意使用 10px、14px、18px 等无体系间距。
- 大面积红色、紫色渐变、发光、毛玻璃或装饰性阴影。
- 卡片套卡片、所有组件统一大圆角、多个主按钮竞争。
- 只实现成功态，不处理加载、空、错误和部分失败。
- 学生端展示风险评分、后台报告或 Agent 决策链路。
- 为一次性页面过早抽象通用组件或引入新依赖。
